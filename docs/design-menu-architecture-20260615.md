# 菜单架构重构设计文档

**版本**：v1.0
**日期**：2026-06-15
**状态**：已讨论定稿 / 待实施

---

## 1. 背景

当前系统菜单已经完成动态化，菜单由后端 `sys_menu` 和角色绑定控制。随着报价接入、价格源、自制件价格、CMS/U9 数据、BOM 结算等能力增加，当前菜单出现了几个问题：

1. 财务日常入口、管理员规则配置、底层数据查询、技术排障入口混在一起。
2. 一些菜单名称偏系统内部实现，例如“报价产品 BOM 准备”“实时成本计算”“价格准备”，普通财务容易误解。
3. 一些功能还不成熟或更适合作为流程内部页面，不适合作为主菜单暴露。
4. `BOM 结算规则`、`价格变量配置` 等影响核算逻辑的配置，不应给普通财务开放。
5. CMS/U9 当前主要是导入和查询业务数据，不应提前拆出“接入监控”类菜单。

本设计文档用于记录本次菜单架构讨论结果。后续如果菜单调整出现问题，可以根据本文还原当时的设计意图、菜单归类和隐藏策略。

## 2. 设计原则

1. **菜单只放稳定入口**：未成熟、仅流程内部使用、容易误解的功能不放主菜单。
2. **按用户心智分类**：财务看到业务流程，管理员看到规则配置，数据管理员看到底层数据。
3. **报价流程内聚**：价格类型确认、价格准备、发起核算、核算结果查看都从报价单接入详情进入，不单独暴露成菜单。
4. **按数据来源归类基础数据**：U9 数据归 U9，CMS 数据归 CMS。
5. **规则和结果分开**：规则配置放规则配置；规则跑出来的明细结果放基础数据或报价详情。
6. **当前没有完整功能的未来菜单先不放**：例如 U9/CMS/OA 接入监控，等真正有同步监控、异常重试、接口日志后再加。

## 5. 不进入主菜单的功能

以下功能当前阶段不作为主菜单暴露，但可以保留页面、路由和接口。

| 功能 | 处理方式 | 原因 |
|---|---|---|
| 报价产品 BOM 准备 | 从主菜单隐藏，作为报价单接入详情中的状态/步骤 | 财务不应理解为自己要“准备 BOM”，容易歧义 |
| 价格准备 | 从主菜单隐藏，放到报价单接入详情流程中 | 属于报价单核算过程，不应绕过报价单单独操作 |
| 实时成本计算 | 从主菜单隐藏，放到报价单接入详情流程中 | 属于产品行核算动作，不是独立工作入口 |
| 包装组件价格 | 暂时隐藏 | 当前主要用于 BOM 补录参考，逻辑未完全成熟，容易被误认为正式价格源 |
| BOM 补录任务 | 当前阶段先不放普通主菜单 | 后续有技术员协同/OA 待办后，可单独设计协同处理菜单 |
| 补录 BOM 版本 | 当前阶段先不放普通主菜单 | 同上 |
| 接入流水 | 视角色保留为管理员辅助入口，不放普通财务主菜单 | 偏接入记录/排查，不是财务日常入口 |
| 数据接入管理 | 当前不新增 | 尚无完整 U9/CMS/OA 接入监控、重试、接口日志功能 |
| U9 接入监控 | 当前不新增 | 未来有同步任务监控后再设计 |
| CMS 接入监控 | 当前不新增 | CMS 当前先作为数据导入/查询入口 |
| OA 接入监控 | 当前不新增 | 未来接 OA 推送、回执、异常重试后再设计 |

## 7. 角色可见范围

### 7.1 管理员

管理员可见全部目标菜单：

```text
报价需求
价格源管理
基础数据管理
核算与结账
规则配置
系统管理
```

管理员用于系统维护、规则配置、数据排查和权限管理。

### 7.2 普通财务

普通财务建议只开放：

```text
报价需求
价格源管理
基础数据管理
核算与结账
```

普通财务不开放：

```text
规则配置
系统管理
```

普通财务不要直接看到 `BOM 结算规则`、`价格变量配置`、`报价单基价映射规则` 等影响系统核算逻辑的配置。

### 7.3 财务主管 / 价格维护人员

可在普通财务基础上追加：

```text
规则配置
```

但是否开放全部规则配置，需要按实际管理职责再细分。

### 7.4 技术协同人员

当前阶段先不新增主菜单。后续 BOM 补录/OA 待办成熟后，可以新增：

```text
协同处理
├─ BOM 补录任务
└─ 补录 BOM 版本
```

该菜单只给技术员、协同处理人员、管理员，不给普通财务。

## 8. 关键业务解释

### 8.1 为什么报价需求只保留两个子菜单

财务的报价主路径应该非常简单：

```text
报价单导入 -> 报价单接入 -> 进入报价单详情 -> 完成价格类型、价格准备、核算、查看结果
```

因此报价需求只保留：

```text
报价需求
├─ 报价单导入
└─ 报价单接入
```

其他状态和动作放在报价单详情页中，不放主菜单。

### 8.2 为什么 BOM 结算明细不放规则配置

`BOM 结算规则` 是规则。

`BOM 结算明细` 是规则执行后的结果明细，用于查看某个产品、某个月、某个 BOM 版本最终被核算使用的明细数据。

处理链路是：

```text
U9 原始 BOM
  ↓
BOM 结算规则
  ↓
BOM 结算明细
  ↓
报价单成本核算
```

因此：

- `BOM 结算规则` 放 `规则配置`
- `BOM 结算明细` 放 `基础数据管理 / U9 数据`
- 普通财务主要在报价单详情中查看对应产品的 BOM 结算明细

### 8.3 为什么 CMS 回收废料映射放 CMS 数据

`CMS 回收废料映射` 虽然影响自制件废料抵扣，但它的数据来源和业务含义属于 CMS。用户查它时，通常是在确认 CMS 中某个自制件/原材料是否有对应废料物料。

因此它放：

```text
基础数据管理 / CMS 数据 / CMS 回收废料映射
```

不放：

```text
规则配置
核算基础配置
```

### 8.4 为什么当前不放数据接入管理

当前系统没有完整的 U9/CMS/OA 接入监控能力，例如：

- 同步任务列表
- 成功/失败数量
- 接口异常原因
- 失败重试
- 回执记录
- 接入审计

因此当前不新增：

```text
数据接入管理
├─ U9 接入监控
├─ CMS 接入监控
└─ OA 接入监控
```

未来真正有接口同步、异常重试、接口日志需求时，再设计该一级菜单。

## 9. 实施建议

建议分两步实施，降低风险。

### 9.1 第一步：仅调整菜单显示和归类

1. 在 `sys_menu` 中调整菜单名称、父子关系、排序。
2. 对当前不进入主菜单的页面设置 `visible = 1` 或只作为隐藏路由保留。
3. 保持页面、路由、接口不删除。
4. 先用管理员账号验证全部目标菜单可访问。

### 9.2 第二步：按角色收敛权限

1. 普通财务隐藏 `规则配置` 和 `系统管理`。
2. 普通财务隐藏流程内部入口：`价格准备`、`实时成本计算`、`报价产品 BOM 准备`。
3. 财务主管按需开放规则配置。
4. 管理员保留全部。

## 10. 回滚方案

如果菜单调整后出现问题，优先按以下方式回滚：

1. 不回滚代码，先恢复 `sys_menu.parent_id`、`menu_name`、`order_num`、`visible`。
2. 恢复角色菜单绑定 `sys_role_menu`。
3. 如果只是某个菜单看不到，优先检查：
   - `visible`
   - `status`
   - `parent_id`
   - `sys_role_menu` 是否绑定
   - `component` 是否仍指向正确页面
4. 如果需要恢复到调整前，可以按本文第 3 节“当前菜单结构快照”重建父子层级。

实施前建议导出以下快照：

```text
sys_menu 全量
sys_role_menu 全量
/api/v1/auth/routers 返回结果
管理员侧边栏截图
普通财务侧边栏截图
```

## 11. 验收标准

菜单调整完成后，需要满足：

- [ ] 管理员可见 6 个一级菜单：报价需求、价格源管理、基础数据管理、核算与结账、规则配置、系统管理
- [ ] 报价需求下只有报价单导入、报价单接入
- [ ] 普通财务不可见规则配置、系统管理
- [ ] 普通财务不可见价格准备、实时成本计算、报价产品 BOM 准备
- [ ] U9 数据下包含料品主档、U9 BOM 原始数据、BOM 层级树查看、BOM 结算明细、BOM 副产品档案、包装组件结构
- [ ] CMS 数据下包含 CMS 导入、计划成本原始数据、车间料工费原始数据、科目成本原始数据、CMS 公共生效来源、CMS 回收废料映射
- [ ] BOM 结算规则在规则配置下
- [ ] BOM 结算明细不在规则配置下
- [ ] 包装组件价格不作为主菜单暴露
- [ ] 报价单接入详情页仍能完成价格类型、价格准备、发起核算、查看成本结果

---

## 12. 最终结论

当前阶段的菜单目标不是做得多，而是让财务用户能清楚完成主流程：

```text
导入报价单 -> 接入报价单 -> 确认价格类型 -> 价格准备 -> 成本核算 -> 查看已核算结果
```

主菜单只承载稳定、清晰、可独立完成任务的入口。流程步骤、半成熟能力、技术排查入口、未来接口监控都不提前暴露给普通用户。
